# AGENTS.md

## Контекст проекта

Проект `ANPR-System-v0.8_web` — это **web-only standalone система** автоматического распознавания номерных знаков.

Desktop-продуктового пути больше нет.
Внешний видеосервер не должен быть обязательной частью архитектуры.
Live preview должен собираться и отдаваться **внутри самого проекта**.

---

## Главная цель текущего этапа

Перевести проект в состояние, где:
- распознавание и live preview работают согласованно;
- preview не зависит от MediaMTX / go2rtc / внешнего SFU / внешнего WHEP-сервера;
- основным live transport является встроенный **HLS**;
- UI показывает видео активных каналов без ручной подстройки `localhost` и без скрытых ошибок.

---

## Обязательные архитектурные правила

1. Не возвращать desktop UI, desktop lifecycle и desktop feature path.
2. Не переносить decode / inference / OCR / CV в браузер.
3. Сохранять независимую обработку каналов.
4. Не делать внешний медиасервер обязательным компонентом.
5. Не строить продуктовую архитектуру вокруг WebRTC adapter к внешнему серверу.
6. Основной live preview должен быть реализован как встроенный FFmpeg -> HLS path.
7. Lifecycle канала и lifecycle preview должны быть согласованы.
8. Ошибки video pipeline нельзя скрывать пустыми `catch` или полным подавлением stderr.
9. Все изменения API, конфигов и структуры директорий документировать в `README.md`.
10. Все user-facing тексты, README, PR summary и комментарии к архитектурным изменениям — на русском.

## Разрешённые целевые решения

### Допустимо
- оставить `apps/video_gateway` как внутренний сервис проекта;
- координировать preview через `apps/api`;
- использовать FFmpeg для генерации HLS;
- использовать `hls.js` в браузере;
- добавлять health/status/diagnostics endpoints;
- улучшать contracts через Pydantic;
- убирать устаревшие WebRTC-эндпоинты из продуктового пути.

### Недопустимо
- добавлять обязательный MediaMTX, go2rtc или иной внешний media server;
- требовать отдельный ручной запуск preview оператором как основную модель;
- хардкодить browser-facing base URL как `http://localhost:...`;
- молча игнорировать ошибки preview;
- ломать существующие API без необходимости и без документации.

## Приоритеты реализации

### Приоритет 1 — работоспособность standalone preview
Нужно добиться, чтобы активный канал:
- запускал ANPR runtime;
- запускал HLS preview;
- был виден в UI.

### Приоритет 2 — корректный lifecycle
Нужно связать:
- `POST /api/channels/{id}/start`
- `POST /api/channels/{id}/stop`
- `POST /api/channels/{id}/restart`

с соответствующим состоянием video preview.

### Приоритет 3 — диагностика
Нужно сделать понятными ошибки:
- ffmpeg not found;
- source open failed;
- playlist was not created;
- unsupported source type;
- gateway unavailable;
- preview start timeout.

### Приоритет 4 — UX и конфигурация UI
Нужно убрать хардкод и silent failure:
- не использовать `localhost:8091` как обязательный browser URL;
- не скрывать ошибки пустыми `catch`;
- показывать статус preview в UI.

## Правила для кода

1. Делать небольшие, проверяемые изменения.
2. Предпочитать явные типы и Pydantic-модели.
3. Не дублировать конфигурацию без необходимости.
4. Если меняется контракт API — обновить README.
5. Если меняется структура проекта — обновить README.
6. Если меняется архитектурное решение — обновить README и этот `AGENTS.md`.
7. По возможности сохранять backward compatibility существующих endpoint-ов.
8. При невозможности сохранить совместимость — явно отмечать breaking changes.

## Что именно нужно изменить в проекте

### В `apps/api`
- связать lifecycle ANPR и lifecycle preview;
- при старте канала запускать preview автоматически или через единый orchestration layer;
- при stop/restart синхронно управлять обоими контурами;
- по возможности отдавать UI корректный preview base/config.

### В `apps/video_gateway`
- оставить только standalone streaming path;
- удалить или деактивировать обязательную зависимость от внешнего WebRTC/media server;
- добавить внятные diagnostics;
- не считать старт успешным, пока playlist реально не создан.

### В `apps/web`
- убрать жёсткий внешний `localhost` как базу для preview;
- показывать ошибки видео;
- не делать пустые silent catch там, где ломается preview;
- сохранить простой операторский UX.

### В `infra/docker-compose.yml`
- удалить внешний медиасервер из обязательного product path;
- оставить только сервисы самого проекта и PostgreSQL;
- не передавать WebRTC env как обязательную конфигурацию.

## Критерии завершения задачи

Изменение считается завершённым, когда:
1. `docker compose up --build` поднимает standalone-схему без внешнего видеосервера.
2. UI показывает видео активного канала.
3. События распознавания и preview работают одновременно.
4. Видео не зависит от ручного browser-side `localhost`.
5. Ошибки preview видимы и диагностируемы.
6. README и `AGENTS.md` обновлены и соответствуют фактической архитектуре.
